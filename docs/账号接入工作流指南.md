# 账号接入工作流指南

文档版本：v1.0
编写日期：2026-06-09
适用范围：`gen2api/media2api` 的 Web Cookie 转 API、Agent Provider 转 API、订阅清单导入和图片/视频生成反代连接器。

## 1. 目标

平台需要把 GitHub 开源社区中调研到的网页 Web Cookie 转 API、CLI/Agent Provider 转 API、订阅清单和媒体任务代理项目，转化为可操作的 provider 账号接入路径。新的账号接入工作流提供一个接近 sub2api 的入口：用户选择 provider 后，主流程只围绕两类资源展开：导入 Web cookie/session，或登记 Agent Provider profile。connector/runner base URL 只是执行层配置，只有对应项目明确要求服务地址时才填写。

核心接口：

| 接口 | 用途 |
| --- | --- |
| `GET /v1/admin/account-setup-workflows` | 查看所有 provider 的接入状态摘要 |
| `GET /v1/admin/account-setup-workflows/{provider_id}` | 查看单个 provider 的详细接入计划 |
| `POST /v1/admin/account-setup-workflows/{provider_id}/run` | 执行接入工作流的一个步骤 |
| `POST /v1/admin/account-setup-quickstart` | 自动识别 Web Cookie / Agent Provider / subscription 路径，一次请求完成计划、安装、导入和预检 |

## 2. 工作流步骤

| step | 说明 | 是否写库 |
| --- | --- | --- |
| `plan` | 只查看当前 provider 的模板、账号、资源清单源、授权资源会话、预检和下一步建议 | 否 |
| `project_blueprint` | 基于某个 `OpenSourceConnectorProject` 生成项目级连接器蓝图 | 否 |
| `apply_blueprint` | 应用项目级蓝图，安装 provider/account/mapping，可选创建资源清单源 | 取决于 `dry_run` |
| `apply_manifest` | 直接根据当前表单安装 external connector manifest | 取决于 `dry_run` |
| `start_authorized_session` | 为 delegated OAuth、Web session assist 或 Agent Provider 授权助手创建授权资源会话；`start_oauth` 仅作为旧请求兼容别名 | 取决于 `dry_run` |
| `create_subscription_source` | 兼容旧 step 名：创建长期资源清单源，适用于资源清单 URL / 批量账号导入 | 取决于 `dry_run` |
| `sync_subscription_source` | 兼容旧 step 名：执行一次资源清单源导入 | 取决于 `dry_run` |
| `production_sync` | 串联导入、能力同步、健康检查、额度同步、preflight 和可选验收 | 取决于 `dry_run` |
| `preflight` | 对 provider/account 运行外部连接器生产预检 | 否 |
| `account_acceptance` | 对账号执行外部验收，可选真实样本 | 取决于验收参数 |

## 3. 推荐操作路径

1. 打开 `/admin?admin_key=dev-admin-key`，进入“开源连接器”区域。
2. 选择 provider，例如 `openai_image`、`gemini`、`qwen`、`grok`、`jimeng`、`kling`、`midjourney`。
3. 点击“账号工作流”，查看 provider 是否已安装、是否有可用账号、是否有长期资源清单源、是否通过预检。
4. 如果是 Web Cookie provider，填写 cookie header、JSON cookie jar、浏览器导出片段或 `secret://...`；平台保存时必须转为 encrypted secret/ref。
5. 如果是 Agent Provider，填写 `agent://...`、Agent profile、credential cache、agent runtime endpoint、工作目录和隔离策略；CLI/OAuth/MCP 材料只作为 Agent Provider 内部来源。
6. 如果该 provider 对应项目明确要求 sidecar/runner 服务地址，再填写可选执行器 Base URL；这是执行层地址，不是资源本体。
7. 先选择 `apply_manifest` 且 `dry_run=true`，确认 manifest、account 和 mapping 预览。
8. 再选择 `apply_manifest` 且 `dry_run=false`，正式安装 provider 和账号。
9. 如 provider 需要批量账号清单，选择 `create_subscription_source`，填写资源清单 URL 后创建长期同步入口。
10. 运行 `preflight`，确认 connector/agent 支持的图片/视频操作与模型能力完整。
10. 在明确额度和风险后运行 `account_acceptance`，真实样本建议从 `max_samples=1` 开始。

### 3.1 快速接入路径

如果已经拿到了 Web Cookie/session、Agent Provider profile、账号引用或资源清单，可以直接使用 quickstart。它会根据输入自动选择路径：

- 有 `subscription_url`、`subscription_content` 或 `accounts` 时走资源清单账号导入；旧字段名仅为兼容。
- 需要 Web session assist、device login 或 Agent Provider profile capture 且尚无 `credential_ref` 时发起兼容的授权资源会话。
- 其它情况走外部 connector manifest 安装。

```powershell
curl -X POST `
  -H "Authorization: Bearer dev-admin-key" `
  -H "Content-Type: application/json" `
  http://127.0.0.1:8080/v1/admin/account-setup-quickstart `
  -d "{
    \"provider_id\":\"gemini\",
    \"mode\":\"auto\",
    \"resource_type\":\"agent_provider\",
    \"credential_ref\":\"agent://providers/gemini/acct_01\",
    \"resource_profile\":{\"workspace_policy\":\"isolated\"},
    \"account_id\":\"acct_gemini_01\",
    \"operations\":[\"text_to_image\",\"image_edit\",\"text_to_video\",\"image_to_video\"],
    \"supported_provider_models\":[\"nano-banana-pro\",\"veo-3.1\"],
    \"dry_run\":true
  }"
```

确认 `steps`、`action_items` 和 `preflight` 后，把 `dry_run` 改为 `false`。如果返回授权会话，先完成返回的 `authorized_resource_session`；旧响应字段 `oauth_session` 只表示兼容别名，再运行 quickstart 或 `account_acceptance`。

## 4. 凭据引用规范

平台不要求用户把敏感凭据写进文档或 UI 明文。推荐只保存引用：

| 引用前缀 | 适用场景 |
| --- | --- |
| `secret://{secret_id}` | 平台 secret 存储中的加密 Web cookie、session、cookie jar 或托管 agent credential |
| `agent://providers/{provider}/acct_01` | Agent Provider profile、credential cache 或 runtime 引用 |

兼容导入时仍可能解析 `vault://`、`websession://`、`oauth://`、`cli://`、`mcp://`、`endpoint://`、`subscription://`、`env://` 等旧前缀，但它们不得作为新表单的平级资源类型展示。导入完成后必须沉淀为 `secret://` 或 `agent://`，执行器地址另存为执行层配置。

`secret://...` 引用会按 active secret 的真实密文复验平台字段；例如 Luma 必须能解析出 `LUMA_API_KEY`，Gemini 必须能解析出 `GEMINI_CREDENTIALS`、`GOOGLE_APPLICATION_CREDENTIALS`、`gemini_oauth_creds_base64` 或 `gemini_oauth_creds_file` 之一。`plain://...` 这类旧 inline 引用会先按明文材料校验，只有带真实字段名的 JSON 或 `.env` 形态才能进入托管流程。

## 5. curl 示例

查看账号工作流：

```powershell
curl -H "Authorization: Bearer dev-admin-key" `
  http://127.0.0.1:8080/v1/admin/account-setup-workflows/gemini
```

dry-run 安装 manifest：

```powershell
curl -X POST `
  -H "Authorization: Bearer dev-admin-key" `
  -H "Content-Type: application/json" `
  http://127.0.0.1:8080/v1/admin/account-setup-workflows/gemini/run `
  -d "{
    \"step\":\"apply_manifest\",
    \"resource_type\":\"agent_provider\",
    \"auth_method\":\"agent_provider_credential\",
    \"credential_ref\":\"agent://providers/gemini/acct_01\",
    \"resource_profile\":{\"workspace_policy\":\"isolated\"},
    \"dry_run\":true
  }"
```

正式安装 manifest：

```powershell
curl -X POST `
  -H "Authorization: Bearer dev-admin-key" `
  -H "Content-Type: application/json" `
  http://127.0.0.1:8080/v1/admin/account-setup-workflows/gemini/run `
  -d "{
    \"step\":\"apply_manifest\",
    \"resource_type\":\"agent_provider\",
    \"auth_method\":\"agent_provider_credential\",
    \"credential_ref\":\"agent://providers/gemini/acct_01\",
    \"resource_profile\":{\"workspace_policy\":\"isolated\"},
    \"dry_run\":false
  }"
```

创建资源清单源：

```powershell
curl -X POST `
  -H "Authorization: Bearer dev-admin-key" `
  -H "Content-Type: application/json" `
  http://127.0.0.1:8080/v1/admin/account-setup-workflows/gemini/run `
  -d "{
    \"step\":\"create_subscription_source\",
    \"source_id\":\"gemini_source_01\",
    \"source_name\":\"Gemini resource list source\",
    \"subscription_url\":\"https://resource-list.example/gemini/accounts.json\",
    \"dry_run\":false
  }"
```

发起并完成授权资源会话：

该示例默认不包含 `provider_base_url`。如果平台的 `/v1/admin/platform-input-conformance` 返回 `runtime_base_url_allowed=true`，且对应开源项目确实要求外部 agent wrapper 地址，才额外加入该字段。

```powershell
curl -X POST `
  -H "Authorization: Bearer dev-admin-key" `
  -H "Content-Type: application/json" `
  http://127.0.0.1:8080/v1/admin/authorized-resource-sessions `
  -d "{
    \"provider_id\":\"gemini\",
    \"auth_method\":\"agent_provider_credential\",
    \"resource_type\":\"agent_provider\",
    \"supported_operations\":[\"text_to_image\",\"text_to_video\"],
    \"supported_provider_models\":[\"nano-banana-pro\",\"veo-3.1\"]
  }"
```

如果授权助手已经返回完成 JSON，可直接完成并创建账号：

```powershell
curl -X POST `
  -H "Authorization: Bearer dev-admin-key" `
  -H "Content-Type: application/json" `
  http://127.0.0.1:8080/v1/admin/authorized-resource-sessions/{session_id}/complete `
  -d "{
    \"connector_response\":{
      \"status\":\"completed\",
      \"account\":{
        \"id\":\"acct_gemini_01\",
        \"label\":\"Gemini production account 01\",
        \"credential_ref\":\"agent://providers/gemini/acct_01\",
        \"auth_method\":\"agent_provider_credential\",
        \"resource_type\":\"agent_provider\",
        \"resource_profile\":{\"workspace_policy\":\"isolated\"},
        \"supported_operations\":[\"text_to_image\",\"text_to_video\"],
        \"supported_provider_models\":[\"nano-banana-pro\",\"veo-3.1\"],
        \"concurrency_limit\":2,
        \"plan\":\"pro\"
      }
    },
    \"create_account\":true
  }"
```

如果授权助手返回的是一次性材料而不是 `credential_ref`，必须按对应开源项目的字段名提交 `credential_value`。平台会先校验字段，再转成 `secret://...`：

```powershell
curl -X POST `
  -H "Authorization: Bearer dev-admin-key" `
  -H "Content-Type: application/json" `
  http://127.0.0.1:8080/v1/admin/authorized-resource-sessions/{session_id}/complete `
  -d "{
    \"credential_value\":{\"apiKey\":\"useapi-key\"},
    \"auth_method\":\"agent_provider_credential\",
    \"resource_type\":\"agent_provider\",
    \"supported_operations\":[\"text_to_video\"],
    \"supported_provider_models\":[\"runway-gen4\"],
    \"create_account\":true
  }"
```

如果完成会话、callback、账号 onboarding 或批量导入直接提交 `secret://...`，后端也会读取 active secret 并复用同一套字段校验；因此不能先创建一个 raw token secret 再用引用绕过平台要求。

授权助手也可以主动回调：

```powershell
curl -X POST `
  -H "Authorization: Bearer dev-admin-key" `
  -H "Content-Type: application/json" `
  http://127.0.0.1:8080/v1/admin/authorized-resource-sessions/{session_id}/callback `
  -d "{\"status\":\"completed\",\"credential_ref\":\"agent://providers/gemini/acct_01\"}"
```

运行生产预检：

```powershell
curl -X POST `
  -H "Authorization: Bearer dev-admin-key" `
  -H "Content-Type: application/json" `
  http://127.0.0.1:8080/v1/admin/account-setup-workflows/gemini/run `
  -d "{
    \"step\":\"preflight\",
    \"operations\":[\"text_to_image\",\"image_to_image\",\"text_to_video\",\"image_to_video\"],
    \"dry_run\":true
  }"
```

## 6. workflow 与蓝图字段对齐

`GET /v1/admin/account-setup-workflows/{provider_id}` 和项目蓝图响应必须返回 `input_requirements`、`dynamic_profile_fields` 与 `resource_profile_template`。如果当前 provider 的开源项目画像声明了必填字段，但请求没有提供对应 `resource_profile`，workflow/blueprint 必须返回 `required_platform_profile_inputs` action item，而不是让用户只看到泛化的 `Credential Ref` 或 `base_url`。

管理台“开源连接器”面板会按当前 provider 动态渲染这些字段。生成蓝图时使用 `/blueprint/apply` 的 `dry_run=true` 发送结构化 payload，因此 Midjourney 这类平台会随请求携带 `guild_id`、`channel_id` 等 profile 字段；Gemini/Qwen 这类平台会携带 Agent profile/runtime 相关字段。`base_url` 仍只在 `runtime_base_url_allowed=true` 且项目确实要求外部 runner/sidecar 时显示和提交。

`/v1/admin/external-connector-manifest-template` 也必须返回同一组字段画像。模板中的 `resource_profile` 只允许使用 `<required>` / `<optional>` 作为占位提示，后端校验不得把这些占位符当作真实输入；例如 Midjourney 模板默认应使用 `secret://...` Web Cookie 引用，并要求用户替换 `guild_id`、`channel_id` 后才能正式 apply。

## 7. 生产准入

账号进入真实图片/视频生成流量前，必须满足：

1. provider 已安装且状态为 `active`。
2. 至少一个 active account 的 `credential_ref` 可用。
3. provider/account mapping 覆盖当前源码中声明的图片、视频和多模态模型类型。
4. sidecar 或 connector base URL 可达。
5. provider health check 通过。
6. capabilities sync 成功。
7. quota sync 或预算控制可用。
8. external connector preflight 无缺失操作。
9. account acceptance 至少一个真实样本通过。
10. 高风险 Web/session 项目完成人工审核、license 审核和并发限制。

## 8. 与 sub2api 的对齐点

该工作流把 Web cookie/session 和 Agent Provider profile 抽象成统一账号来源，把“网页登录态、CLI/OAuth cache、MCP/agent config”等原始材料归入这两类可治理资源，把“图片/视频生成模型”抽象成 provider capabilities 和 model mapping。最终用户看到的是可选 provider、可导入 Web cookie、可登记 Agent Provider、仅在项目要求时填写执行器地址、可 dry-run 的接入按钮，以及明确的下一步提示，而不是散落的脚本和仓库说明。

批量账号导入默认会自动创建缺失的 model mappings。导入器会读取 sub2api/CLIProxy/AIClient2API 风格导出里的 `models[].operations`，将 `t2i`、`edit`、`t2v`、`i2v` 映射到平台内置逻辑模型。这样账号导入后可以直接进入路由预览和生产预检；如需纯账号导入，可传入 `auto_create_mappings=false`。

新增 smoke 脚本：

```powershell
.\.venv\Scripts\python.exe scripts\account_setup_workflow_smoke_test.py

```

该脚本使用临时 SQLite 数据库，自动种入管理员 API key 和一个 Gemini 开源连接器项目，覆盖：

1. 工作流列表和单 provider 详情。
2. 项目级蓝图生成。
3. manifest dry-run 和正式安装。
4. 账号工作流从 action required 进入 ready。
5. 资源清单源 dry-run 和正式创建。
6. external connector preflight。
7. `/admin` 控件、operator workbench 和 delivery package 可见性。
8. 授权资源会话 completion 和 callback 自动创建账号。
9. sub2api/CLIProxy/AIClient2API 风格嵌套账号导出可预览并导入。
10. 参考外部 sidecar 可以以 `secret://` 服务凭据 + `agent://` 账号引用方式覆盖文生图、图像编辑、文生视频、图生视频，并通过生产预检和账号样例验收。
11. quickstart 可以用一个请求覆盖 manifest dry-run/live 安装、资源清单导入自动映射和授权会话 dry-run。

反代 sidecar 的 header、payload、轮询和结果返回契约见 `docs/反代连接器运行时契约.md`，并可用 `scripts/connector_smoke_test.py` 验证。
