# 反代内核运行时接入指南

本文约束已定型反代内核如何从“仓库参考”进入“可运行执行器”状态。

## 原则

1. 优先使用 release 中已经构建好的二进制或压缩包。
2. 完整仓库源码只在 release 不存在、能力不透明、需要协议参考或需要重写 adapter 时拉到 `source-repo/`。
3. 受控执行器必须固定版本、固定 SHA256。
4. 受控执行器只能监听 `127.0.0.1`、`localhost` 或 `::1`。
5. 不能把第三方反代项目作为公网长期服务直接暴露给用户。
6. 平台仍然负责公开 API、账号池、用户 API Key、审计、计费和资产入库。

## 管理接口

| 接口 | 用途 |
|---|---|
| `GET /v1/admin/proxy-kernels` | 查看全部定型内核、账号状态、runtime 状态和阻塞项 |
| `GET /v1/admin/proxy-kernels/{provider_id}` | 查看单个 provider 的内核状态 |
| `POST /v1/admin/proxy-kernels/{provider_id}/release-probe` | 从 GitHub release 探测可用资产 |
| `POST /v1/admin/proxy-kernels/{provider_id}/install-release` | 下载并校验指定 release 资产 |
| `POST /v1/admin/proxy-kernels/{provider_id}/register-runtime` | 登记已启动的 loopback runtime，并同步 provider `base_url` |
| `POST /v1/admin/proxy-kernels/{provider_id}/start-runtime` | 使用已校验资产启动受控本地执行器，并同步 provider `base_url` |
| `POST /v1/admin/proxy-kernels/{provider_id}/stop-runtime` | 停止平台启动的受控执行器 |
| `GET /v1/admin/proxy-kernels/{provider_id}/process` | 查看受控执行器 PID、命令、日志路径和运行状态 |
| `GET /v1/admin/proxy-kernels/{provider_id}/logs` | 查看 stdout/stderr 最新日志 |
| `POST /v1/admin/proxy-kernels/{provider_id}/clear-runtime` | 停止受控执行器，清除 runtime 登记和 provider `base_url` |
| `GET /v1/admin/proxy-kernels/{provider_id}/source-repo` | 查看定型仓库源码参考的本地状态 |
| `POST /v1/admin/proxy-kernels/{provider_id}/source-repo/sync` | 将定型仓库同步到 `source-repo/`，仅用于协议参考、本地构建或 adapter 重写 |

管理后台也提供同样的可视化路径：

1. 打开 `/admin`。
2. 进入“反代内核”。
3. 在“运行状态”查看定型内核、账号/runtime/blocker。
4. 在“启动执行器”先探测 release，填写 asset 名称和 SHA256 后点击“安装 Release”。
5. 安装成功后，页面会自动填入资产路径和 SHA256；再填写 loopback base URL 和启动命令。
6. 在“日志与停止”查看 stdout/stderr、停止 runtime 或清除登记。
7. 只有 release 资产不可用或需要协议参考时，进入“源码参考”同步定型仓库。

## 安装 release

安装时必须提供 `expected_sha256`。没有 hash 的 release 资产只能探测，不能登记为已校验安装。

```bash
curl -X POST "$MEDIA2API_BASE_URL/v1/admin/proxy-kernels/openai_web_session/release-probe" \
  -H "Authorization: Bearer $MEDIA2API_API_KEY"
```

```bash
curl -X POST "$MEDIA2API_BASE_URL/v1/admin/proxy-kernels/openai_web_session/install-release" \
  -H "Authorization: Bearer $MEDIA2API_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "tag_name": "vX.Y.Z",
    "asset_name": "example-linux-amd64.tar.gz",
    "expected_sha256": "64位十六进制sha256"
  }'
```

## 登记 loopback runtime

当你已经用固定版本/hash 的 release 二进制在服务器本地启动执行器后，登记它：

```bash
curl -X POST "$MEDIA2API_BASE_URL/v1/admin/proxy-kernels/openai_web_session/register-runtime" \
  -H "Authorization: Bearer $MEDIA2API_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "base_url": "http://127.0.0.1:19081",
    "version": "vX.Y.Z",
    "binary_path": "/opt/media2api/var/proxy-kernels/openai_web_session/vX.Y.Z/example-linux-amd64",
    "sha256": "64位十六进制sha256",
    "notes": "OAI-WEB-01 verified release"
  }'
```

`register-runtime` 会拒绝非 loopback 地址。通过后，平台会同步更新该 provider 的 `base_config.base_url`，后续能力同步、health check 和媒体调用才会走这个受控执行器。

## 源码参考

源码不是默认运行路径。只有以下情况才同步到 `source-repo/`：

1. GitHub release 没有可用二进制。
2. release asset 不完整或无法确认媒体路由能力。
3. 需要确认 cookie/session、CLI OAuth、任务轮询或资产下载协议。
4. 需要参考逻辑重写平台内 adapter。

查看状态：

```bash
curl "$MEDIA2API_BASE_URL/v1/admin/proxy-kernels/openai_web_session/source-repo" \
  -H "Authorization: Bearer $MEDIA2API_API_KEY"
```

同步定型仓库：

```bash
curl -X POST "$MEDIA2API_BASE_URL/v1/admin/proxy-kernels/openai_web_session/source-repo/sync" \
  -H "Authorization: Bearer $MEDIA2API_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{"ref":"","force":false}'
```

平台只允许同步内核选型表中的仓库，路径固定为 `source-repo/<owner>__<repo>`。如果目录已存在且不是同一个 GitHub 远端，会拒绝同步。

## 受控启动 loopback runtime

如果 release 资产已经放在 `MEDIA2API_PROXY_KERNEL_DIR` 下，并且你掌握该资产的 SHA256，可以让平台直接启动它。启动命令必须是参数数组，不能走 shell；`artifact_path` 必须出现在 `command` 中，避免记录的 hash 和实际启动文件不一致。

```bash
curl -X POST "$MEDIA2API_BASE_URL/v1/admin/proxy-kernels/openai_web_session/start-runtime" \
  -H "Authorization: Bearer $MEDIA2API_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "command": [
      "/opt/media2api/var/proxy-kernels/openai_web_session/vX.Y.Z/example-linux-amd64",
      "--host",
      "127.0.0.1",
      "--port",
      "19081"
    ],
    "base_url": "http://127.0.0.1:19081",
    "artifact_path": "/opt/media2api/var/proxy-kernels/openai_web_session/vX.Y.Z/example-linux-amd64",
    "expected_sha256": "64位十六进制sha256",
    "version": "vX.Y.Z",
    "replace_existing": true
  }'
```

常用检查：

```bash
curl "$MEDIA2API_BASE_URL/v1/admin/proxy-kernels/openai_web_session/process" \
  -H "Authorization: Bearer $MEDIA2API_API_KEY"

curl "$MEDIA2API_BASE_URL/v1/admin/proxy-kernels/openai_web_session/logs?stream=stderr&max_bytes=12000" \
  -H "Authorization: Bearer $MEDIA2API_API_KEY"
```

停止受控执行器：

```bash
curl -X POST "$MEDIA2API_BASE_URL/v1/admin/proxy-kernels/openai_web_session/stop-runtime" \
  -H "Authorization: Bearer $MEDIA2API_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{"grace_seconds":5}'
```

## 状态解释

`GET /v1/admin/proxy-kernels` 的 `blockers` 字段用于告诉用户还缺什么：

| blocker | 含义 |
|---|---|
| `PROVIDER_NOT_INITIALIZED` | 还没有通过账号导入或模板初始化 provider |
| `NO_ACTIVE_ACCOUNT` | 还没有导入真实可用账号材料 |
| `NO_LOOPBACK_RUNTIME` | 还没有登记受控 loopback 执行器或完成原生 adapter 重写 |
| `KERNEL_HASH_NOT_VERIFIED` | 已安装资产的 SHA256 与 expected SHA256 不一致 |
| `KERNEL_PROCESS_STOPPED` | 已由平台登记受控进程，但当前进程未运行 |

只有 provider、账号、runtime/hash 与真实样本验收都通过后，才能把该 provider 标记为生产可用。
