# 反代内核运行时接入指南

本文约束已定型反代内核如何从“仓库参考”进入“可运行执行器”状态。

## 原则

1. 优先使用 release 中已经构建好的二进制或压缩包。
2. 完整仓库源码只在 release 不存在、能力不透明、需要协议参考或需要重写 adapter 时拉到 `source-repo/`。
3. 受控执行器必须固定版本、固定 SHA256。
4. 受控执行器只能监听 `127.0.0.1`、`localhost` 或 `::1`。
5. 不能把第三方反代项目作为公网长期服务直接暴露给用户。
6. 平台仍然负责公开 API、账号池、用户 API Key、审计、计费和资产入库。

管理后台也提供同样的可视化路径：

1. 打开 `/admin`。
2. 进入“反代内核”。
3. 在“运行状态”查看定型内核、账号/runtime/blocker。
4. 优先点击“上线执行向导”，先看当前 provider 的下一步：补路由、导入真实账号、准备 release/source runtime、健康检查、真实样本验收或准备下游 API Key。页面会渲染 6 个固定阶段卡片；卡片按钮可直接跳转到账号导入、启动执行器、用户 API Key 页面，或执行补路由、健康检查、验收 dry-run 等平台可安全处理的动作；底部 JSON 只用于复制和排查。
5. 在“运行状态”点击“查看全部路由计划”，确认全部定型 provider 还缺什么。
6. 点击“补齐全部定型路由”批量创建 provider 与模型映射；它不会创建假账号。
7. 点击“查看运行时交付计划”，确认每个 provider 当前下一步是探测 release、填写 SHA256、启动 loopback、同步 source-repo 还是导入账号。
8. 点击“上线工作台预检”，一次性汇总全量 provider 的路由、release 候选、runtime、账号、健康检查和真实样本验收缺口；默认 dry-run，不下载二进制、不创建假账号。
9. 点击“全量 Release 探测”，确认哪些仓库已有可优先安装的 release 资产，哪些仓库需要 source-repo 兜底。
10. 点击“全量 Hash 候选”，让平台只读取 release 中的小型 checksum/sha256 文件，生成 `install_payload_template`；它不会下载二进制。
11. 点击“批量安装候选计划”，生成全部 provider 的 checksum-resolved 安装计划；该按钮只做 dry-run，不下载二进制。
12. 点击“运行合同矩阵”，确认每个 provider 的图片/视频提交端点、账号引用头、轮询路径和输出解析规则都可由平台适配。
13. 点击“生产就绪矩阵”，查看每个 provider 当前卡在路由、runtime、账号、健康检查还是真实样本验收。
14. 点击“查看全部材料清单”，先确认每个 provider 需要你提供哪些账号/session、release、SHA256、loopback 地址和验收材料。
15. 点击“Loopback 合同自检”，先验证平台自身能把图片/视频请求完整转发给本机反代内核并完成资产入库。
16. 点击“查看全部上线清单”，确认每个 provider 下一步到底是补路由、装 runtime、导账号还是跑真实样本。
17. 在“启动执行器”先探测 release，再点“Hash 候选”；如解析成功，页面会自动回填 release tag、asset 名称和 SHA256。
18. 如果 checksum 候选已经匹配到 preferred release asset，可点击“安装 Hash 候选”；它会下载该资产并继续执行 SHA256 校验。
19. 如果没有自动匹配候选，确认 SHA256 来源可信后手动填写并点击“安装 Release”。
20. 安装成功后，平台会在 SHA256 校验通过后安全解包 `.zip`、`.tar`、`.tar.gz`、`.tgz` 等 release 资产，并记录 `executable_candidates`；页面会优先回填可执行候选路径及候选文件 SHA256。
21. 再填写 loopback base URL 和启动命令；如果没有可执行候选，说明该 release 仍需要人工确认启动入口或进入 `source-repo/` 做协议参考。
22. 启动时保留“启动后健康检查”为“运行”，平台会调用同一个 provider health-check 合同并写入生产就绪证据；如果 runner 启动较慢，可先跳过，稍后点“Runtime 健康检查”。
23. 在“真实验收模式”保持 `Dry run`，点击“运行真实验收”；它会聚合 runtime health、账号验收套件和真实样本条件，但不会调用上游或消耗额度。
24. Dry-run 显示 `ready_to_run` 后，再把模式切成 `Live`，保持 `最大样本数=1`，运行第一条真实图片/视频样本验收。
25. 点击“查看路由计划”确认单个 provider 还缺账号、runtime 还是映射；点击“补齐路由映射”只会创建该 provider 的 `ProviderModelMapping`。
26. 点击“运行合同”读取单个 provider 的标准请求、账号引用头、轮询和输出入库合同。
27. 点击“生产就绪”读取单个 provider 的阶段证据、最近健康检查和真实样本覆盖。
28. 点击“查看材料清单”读取单个 provider 的 `required_materials`、账号字段、runtime 字段和可执行 curl。
29. 点击“查看上线清单”读取单个 provider 的账号材料、runtime 和验收命令。
30. 在“日志与停止”查看 stdout/stderr、停止 runtime 或清除登记。
31. 只有 release 资产不可用或需要协议参考时，进入“源码参考”同步定型仓库。

## 安装 release

安装时必须提供 `expected_sha256`。没有 hash 的 release 资产只能探测，不能登记为已校验安装。安装成功后，如果资产是常见压缩包，平台会在受控目录内安全解包，禁止路径逃逸和链接成员，并记录可执行候选及候选文件自身的 SHA256，供 `start-runtime` 使用。

先用 dry-run 确认本次会覆盖哪些定型仓库；它不访问 GitHub、不下载资产：

```bash
curl "$MEDIA2API_BASE_URL/v1/admin/proxy-kernels/release-probe-matrix?dry_run=true" \
  -H "Authorization: Bearer $MEDIA2API_API_KEY"
```

真实批量探测会访问各定型仓库的 GitHub release，但仍然只读、不下载、不启动进程：

```bash
curl "$MEDIA2API_BASE_URL/v1/admin/proxy-kernels/release-probe-matrix" \
  -H "Authorization: Bearer $MEDIA2API_API_KEY"
```

响应中每行的 `install_ready=true` 只代表找到候选 release 资产；实际安装仍必须由操作者选择 `asset_name` 并提供可信来源的 `expected_sha256`。

继续读取 release 中的 checksum 候选。`dry_run=true` 不访问 GitHub；默认调用只下载小型 checksum/sha256 文本，不下载二进制、不安装：

```bash
curl "$MEDIA2API_BASE_URL/v1/admin/proxy-kernels/release-checksum-matrix?dry_run=true" \
  -H "Authorization: Bearer $MEDIA2API_API_KEY"

curl "$MEDIA2API_BASE_URL/v1/admin/proxy-kernels/release-checksum-matrix" \
  -H "Authorization: Bearer $MEDIA2API_API_KEY"

curl "$MEDIA2API_BASE_URL/v1/admin/proxy-kernels/openai_web_session/release-checksums" \
  -H "Authorization: Bearer $MEDIA2API_API_KEY"
```

关键字段：

- `resolved_sha256_candidates`: 已匹配到 release asset 的 SHA256 候选。
- `install_payload_template`: 可复制到 `POST /v1/admin/proxy-kernels/{provider_id}/install-release` 的请求体。
- `unresolved_preferred_assets`: 找到了候选 runtime 资产，但 release 中没有匹配 checksum，需要人工确认 SHA256。
- `policy.downloaded_binaries=false`: 该接口绝不下载二进制。

使用已解析候选安装 release。默认 `dry_run=true` 且 `resolve_release=false` 时只返回计划；设置 `dry_run=false` 后会重新解析 checksum、选择匹配的 preferred asset，并调用同一套 SHA256 校验安装逻辑：

```bash
curl -X POST "$MEDIA2API_BASE_URL/v1/admin/proxy-kernels/openai_web_session/install-release-candidate" \
  -H "Authorization: Bearer $MEDIA2API_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{"dry_run":true,"resolve_release":false}'

curl -X POST "$MEDIA2API_BASE_URL/v1/admin/proxy-kernels/openai_web_session/install-release-candidate" \
  -H "Authorization: Bearer $MEDIA2API_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{"dry_run":false,"resolve_release":true,"allow_non_preferred":false,"force":false}'
```

批量安装候选接口默认也是 dry-run。后台“批量安装候选计划”按钮使用 `dry_run=true`、`resolve_release=true` 生成全量可安装计划，不下载二进制：

```bash
curl -X POST "$MEDIA2API_BASE_URL/v1/admin/proxy-kernels/install-release-candidates" \
  -H "Authorization: Bearer $MEDIA2API_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{"dry_run":true,"resolve_release":true,"provider_ids":[],"allow_non_preferred":false,"continue_on_error":true}'
```

## 运行合同矩阵

运行合同矩阵用于回答“平台会如何调用这个反代内核”。它不访问上游、不下载 release、不启动进程。

```bash
curl "$MEDIA2API_BASE_URL/v1/admin/proxy-kernels/runtime-contract-matrix" \
  -H "Authorization: Bearer $MEDIA2API_API_KEY"

curl "$MEDIA2API_BASE_URL/v1/admin/proxy-kernels/openai_web_session/runtime-contract" \
  -H "Authorization: Bearer $MEDIA2API_API_KEY"
```

关键字段：

- `adapter_contract.endpoints`: 每个 operation 实际提交到 loopback runner 的路径。
- `adapter_contract.task_id_paths`、`status_paths`、`output_paths`: 异步任务 ID、状态和输出资产的解析规则。
- `auth_contract.header_names`: 平台传给 runner 的账号引用头。
- `checks`: 合同是否覆盖 http adapter、端点映射、`/v1/images/*`、`/v1/videos/*`、loopback-only 和禁止官方 SDK/API。
- `state`: 路由、runtime、账号、hash 的当前证据。

## 生产就绪矩阵

生产就绪矩阵用于回答“这个 provider 是否可以直接承接真实用户图片/视频请求”。它只读取当前证据，不自动运行上游调用。

```bash
curl "$MEDIA2API_BASE_URL/v1/admin/proxy-kernels/production-readiness-matrix" \
  -H "Authorization: Bearer $MEDIA2API_API_KEY"

curl "$MEDIA2API_BASE_URL/v1/admin/proxy-kernels/openai_web_session/production-readiness" \
  -H "Authorization: Bearer $MEDIA2API_API_KEY"
```

必须同时满足：

- `routing`: provider 和模型映射已启用。
- `runtime_contract`: 平台到 loopback runner 的调用合同 ready。
- `runtime`: 已登记或启动 loopback runtime，且受控进程未停止。
- `release_or_runtime_artifact`: release/hash 已校验，或已有可追溯 runtime 登记。
- `account`: 至少一个真实可用账号材料已导入。
- `health`: 最近一次 provider health-check 通过。
- `live_acceptance`: 声明操作已有真实 completed job 和可下载资产证据。

```bash
curl -X POST "$MEDIA2API_BASE_URL/v1/admin/proxy-kernels/openai_web_session/release-probe" \
  -H "Authorization: Bearer $MEDIA2API_API_KEY"
```

```bash
curl -X POST "$MEDIA2API_BASE_URL/v1/admin/proxy-kernels/openai_web_session/install-release" \
  -H "Authorization: Bearer $MEDIA2API_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "tag_name": "vX.Y.Z",
    "asset_name": "example-linux-amd64.tar.gz",
    "expected_sha256": "64位十六进制sha256"
  }'
```

## 登记 loopback runtime

当你已经用固定版本/hash 的 release 二进制在服务器本地启动执行器后，登记它：

```bash
curl -X POST "$MEDIA2API_BASE_URL/v1/admin/proxy-kernels/openai_web_session/register-runtime" \
  -H "Authorization: Bearer $MEDIA2API_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "base_url": "http://127.0.0.1:19081",
    "version": "vX.Y.Z",
    "binary_path": "/opt/media2api/var/proxy-kernels/openai_web_session/vX.Y.Z/example-linux-amd64",
    "sha256": "64位十六进制sha256",
    "notes": "OAI-WEB-01 verified release"
  }'
```

`register-runtime` 会拒绝非 loopback 地址。通过后，平台会同步更新该 provider 的 `base_config.base_url`，后续能力同步、health check 和媒体调用才会走这个受控执行器。

## 路由映射

从零清空数据后，平台不会自动生成假账号。执行器安装/登记之后，可以先查看路由计划：

```bash
curl "$MEDIA2API_BASE_URL/v1/admin/proxy-kernels/routing-plan" \
  -H "Authorization: Bearer $MEDIA2API_API_KEY"
```

```bash
    "provider_ids": [],
    "status": "active",
    "enable_mappings": true,
    "priority_offset": 0,

```

如果只想检查单个 provider：

```bash
curl "$MEDIA2API_BASE_URL/v1/admin/proxy-kernels/openai_web_session/routing-plan" \
  -H "Authorization: Bearer $MEDIA2API_API_KEY"
```

如果 `missing_mappings` 或 `NO_ENABLED_MAPPINGS` 存在，可以应用定型模板映射：

```bash
curl -X POST "$MEDIA2API_BASE_URL/v1/admin/proxy-kernels/openai_web_session/apply-routing" \
  -H "Authorization: Bearer $MEDIA2API_API_KEY" \
  -H "Content-Type: application/json" \
```

## 材料准备清单

材料准备清单是只读接口，用来回答“要把这个 provider 接起来，我现在到底还需要给系统什么”。它不会创建 provider、不会导入账号、不会启动 runtime、不会调用上游，也不会消耗任何账号额度。

查看全部定型 provider：

```bash
curl "$MEDIA2API_BASE_URL/v1/admin/proxy-kernels/materials-request" \
  -H "Authorization: Bearer $MEDIA2API_API_KEY"
```

查看单个 provider：

```bash
curl "$MEDIA2API_BASE_URL/v1/admin/proxy-kernels/openai_web_session/materials-request" \
  -H "Authorization: Bearer $MEDIA2API_API_KEY"
```

返回重点字段：

| 字段 | 含义 |
|---|---|
| `required_materials` | 当前真正缺少的材料；区分用户必须提供和平台可自动生成 |
| `message_to_user` | 给操作者看的最短行动清单 |
| `account_materials` | 账号授权材料，如 Web cookie/session、Codex profile、Gemini CLI OAuth、Discord guild/channel |
| `runtime_materials` | release asset、SHA256、loopback base URL、启动命令和 `source-repo/` 兜底策略 |
| `routing_materials` | provider/model mapping 是否已准备；平台可通过 `apply-routing` 生成 |
| `validation_materials` | 真实样本验收需要的 admin key、user key、样本模型和操作范围 |

## 操作者交付包

交付包是给实际操作者看的可提交材料包。它复用材料清单、运行时交付计划、上线清单和生产就绪证据，把“还缺什么”和“把材料提交到哪里”整理成可复制 JSON。该接口只读，不会调用上游、不下载二进制、不创建账号。

```bash
curl "$MEDIA2API_BASE_URL/v1/admin/proxy-kernels/operator-handoff" \
  -H "Authorization: Bearer $MEDIA2API_API_KEY"

curl "$MEDIA2API_BASE_URL/v1/admin/proxy-kernels/openai_web_session/operator-handoff" \
  -H "Authorization: Bearer $MEDIA2API_API_KEY"
```

执行交付包预演，平台会按默认顺序检查并规划：补路由、提交账号材料、安装 release、启动 runtime、健康检查、dry-run 验收。默认 `dry_run=true`，不会下载二进制、不会创建账号、不会调用上游：

```bash
curl -X POST "$MEDIA2API_BASE_URL/v1/admin/proxy-kernels/openai_web_session/operator-handoff/run" \
  -H "Authorization: Bearer $MEDIA2API_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{"dry_run":true}'
```

只有在你显式传入 `dry_run=false`，并且 `account_onboarding`、`install_release`、`source_runtime_setup`、`source_runtime_launcher`、`start_runtime` 等 payload 已经替换掉全部占位符后，平台才会逐步执行。执行顺序可用 `steps` 控制；推荐先只跑 `["apply_routing","runtime_health_check","live_acceptance_dry_run"]` 这类不消耗额度的组合。默认编排顺序包含 `apply_routing`、`submit_account_material`、`install_release`、`source_runtime_setup`、`source_runtime_launcher`、`start_runtime`、`runtime_health_check`、`live_acceptance_dry_run`；如果 `source_runtime_launcher` 成功生成 hash 启动器，后续 `start_runtime` 会自动使用其返回的 `start_payload_template`。

重点字段：

| 字段 | 含义 |
|---|---|
| `operator_questions` | 当前仍需人工填写的账号或 runtime 字段 |
| `submission_templates.account_onboarding_inline_secret` | 直接粘贴账号材料时可用的账号导入 JSON |
| `submission_templates.account_onboarding_secret_ref` | 已经托管为 `secret://` 或 `agent://` 引用时可用的账号导入 JSON |
| `submission_templates.install_release` | release asset 与 SHA256 安装模板 |
| `submission_templates.source_runtime_plan` | source-repo 已同步时的依赖、构建和启动候选计划 |
| `submission_templates.source_runtime_setup` | source-repo 依赖/构建准备模板；命令必须来自 `source-runtime-plan` |
| `submission_templates.source_runtime_launcher` | source-repo hash 启动器生成模板，成功后会产出 `start_runtime` 可用的 payload |
| `submission_templates.start_runtime` | 已校验二进制启动 loopback runtime 的模板 |
| `submission_templates.live_acceptance_dry_run` | 不消耗额度的真实验收预检模板 |
| `submission_templates.live_acceptance` | 消耗真实额度的 live 样本验收模板，建议 `max_samples=1` 起步 |
| `submission_templates.operator_handoff_run` | 可直接提交给 `/operator-handoff/run` 的安全 dry-run 编排模板 |
| `steps` | 从补路由、导账号、装 runtime 到 dry-run/live 验收的顺序 |

材料清单与上线清单的区别：

- 材料清单面向操作者，告诉你“请提供哪些东西”。
- 上线清单面向验收，告诉你“这些东西是否已经满足上线前置条件”。
- 两者都不使用官方 SDK/API，不调用上游，不创建假账号。

后台“可直接用”不是“已经填了账号和 runtime”的同义词。它要求路由、loopback runtime、真实账号、健康检查和每个声明图片/视频操作的 live 样本资产都具备证据；只满足前置条件但未跑真实样本时，会显示为“待真实验收”。

`GET /v1/admin/proxy-kernels` 的摘要也会按这个口径拆分：`needs_route`、`needs_account`、`needs_runtime`、`needs_health` 和 `needs_live_acceptance` 分别指向下一类缺口。只有这些缺口全部清零且 `live_acceptance_ok=true`，`usable/directly_usable` 才会变成 `true`。

部署时默认启用 `MEDIA2API_PROXY_KERNEL_BOOTSTRAP_ROUTES=true`。该开关只初始化定型 provider 和 provider/model 映射，属于平台静态路由配置；它不创建账号、不写入凭据、不生成任务，也不会恢复 mock 数据。`MEDIA2API_SEED_DEFAULTS=false` 时仍可保持平台数据从 0 开始，但路由缺口会自动清零，下一步直接进入账号、runtime 和验收。

## Loopback 合同自检

Loopback 合同自检用于验证平台自身对反代内核的通用调用合同：临时启动一个只监听 `127.0.0.1` 的 runner，临时创建 provider、账号、模型映射和 pricing rule，然后实际跑 `text_to_image`、`image_edit`、`text_to_video` 三类任务。通过后说明平台调度、账号租约、`http_adapter` 转发、返回资产解析和资产入库链路是闭环的。

```bash
curl -X POST "$MEDIA2API_BASE_URL/v1/admin/proxy-kernels/loopback-contract-test" \
  -H "Authorization: Bearer $MEDIA2API_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{"operations":["text_to_image","image_edit","text_to_video"]}'
```

该自检的边界：

- 它不调用任何真实上游平台，也不使用官方 SDK/API。
- 它不替代 OpenAI/Gemini/豆包/Qwen 等真实账号验收。
- 它只证明“平台 -> loopback 反代内核 -> 平台资产入库”这条通用合同可用。
- 运行结束后临时 provider、账号、模型映射和 pricing rule 会被禁用，不作为生产路由使用。

## 受控启动 loopback runtime

如果 release 资产已经放在 `MEDIA2API_PROXY_KERNEL_DIR` 下，并且你掌握该资产的 SHA256，可以让平台直接启动它。启动命令必须是参数数组，不能走 shell；`artifact_path` 必须出现在 `command` 中，避免记录的 hash 和实际启动文件不一致。

```bash
curl -X POST "$MEDIA2API_BASE_URL/v1/admin/proxy-kernels/openai_web_session/start-runtime" \
  -H "Authorization: Bearer $MEDIA2API_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "command": [
      "/opt/media2api/var/proxy-kernels/openai_web_session/vX.Y.Z/example-linux-amd64",
      "--host",
      "127.0.0.1",
      "--port",
      "19081"
    ],
    "base_url": "http://127.0.0.1:19081",
    "artifact_path": "/opt/media2api/var/proxy-kernels/openai_web_session/vX.Y.Z/example-linux-amd64",
    "expected_sha256": "64位十六进制sha256",
    "version": "vX.Y.Z",
    "replace_existing": true,
    "run_health_check": true,
    "fail_on_health_check": false
  }'
```

常用检查：


## 状态解释

`GET /v1/admin/proxy-kernels` 的 `blockers` 字段用于告诉用户还缺什么：

| blocker | 含义 |
|---|---|
| `PROVIDER_NOT_INITIALIZED` | 还没有通过账号导入或模板初始化 provider |
| `PROVIDER_DISABLED` | provider 已存在但未启用，暂不会参与路由 |
| `NO_ENABLED_MAPPINGS` | 还没有启用该 provider 的模板模型映射 |
| `TEMPLATE_MAPPINGS_INCOMPLETE` | 定型模板中还有部分模型映射未写入 |
| `NO_ACTIVE_ACCOUNT` | 还没有导入真实可用账号材料 |
| `NO_LOOPBACK_RUNTIME` | 还没有登记受控 loopback 执行器或完成原生 adapter 重写 |
| `KERNEL_HASH_NOT_VERIFIED` | 已安装资产的 SHA256 与 expected SHA256 不一致 |
| `KERNEL_PROCESS_STOPPED` | 已由平台登记受控进程，但当前进程未运行 |

只有 provider、账号、runtime/hash 与真实样本验收都通过后，才能把该 provider 标记为生产可用。

## 运行时交付计划

运行时交付计划是只读接口，用来回答“这个反代内核现在到底卡在哪一步”。它不会下载 release、不会启动进程、不会同步源码、不会创建账号，也不会调用上游。

查看全部 provider：

```bash
curl "$MEDIA2API_BASE_URL/v1/admin/proxy-kernels/runtime-delivery-plan" \
  -H "Authorization: Bearer $MEDIA2API_API_KEY"
```

查看单个 provider：

```bash
curl "$MEDIA2API_BASE_URL/v1/admin/proxy-kernels/openai_web_session/runtime-delivery-plan" \
  -H "Authorization: Bearer $MEDIA2API_API_KEY"
```

重点字段：

| 字段 | 含义 |
|---|---|
| `next_step` | 当前最该执行的一步，例如 `apply_routing`、`probe_release`、`install_release`、`start_runtime`、`import_account`、`live_acceptance` |
| `state` | 当前证据：路由、release、hash、runtime、进程、账号、source-repo 是否已具备 |
| `runtime.install_payload_template` | 安装 release 时需要填写的 `tag_name`、`asset_name`、`expected_sha256` 模板 |
| `runtime.start_payload_template` | 启动 loopback runtime 的命令数组、base URL、artifact path 和 SHA256 模板 |
| `runtime.source_repo_payload_template` | 只有 release 不可用或需要协议参考时，才使用的 source-repo 同步模板 |
| `commands` | 对应 curl 命令模板，便于复制执行或对照后台表单 |

推荐顺序：

1. `apply_routing`：先把 provider 和模型映射补齐。
2. `probe_release`：优先探测 GitHub release。
3. `release_checksums`：读取 release checksum，优先得到可验证的 SHA256 候选。
4. `install_release_candidate`：如果候选匹配 preferred asset，直接安装并校验。
5. `install_release`：没有候选时，人工填入 release asset 名称和 64 位 SHA256，安装并校验。
6. `start_runtime`：用已校验资产启动只监听 loopback 的执行器。
7. `import_account`：去“授权资源/账号池”导入真实 Web session 或 Agent Provider 材料。
8. `live_acceptance`：运行真实样本验收，确认图片/视频资产真的入库。
