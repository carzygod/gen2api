# 反代内核仓库选型定型文档

日期：2026-06-12

本文根据最终采用清单固化第一阶段仓库选型，作为后续 provider 拆分、账号导入、执行内核接入和验收的准入依据。

## 定型前提

- 禁止官方 SDK / 官方公开 API key 路线。
- 只采用反代、Web session、CLI OAuth/session、客户端本地凭据、订阅转 API 类方案。
- 不内置并长期托管第三方反代项目作为公开服务。
- 允许少量 Go/Node/Python 项目作为固定版本、固定 hash、loopback-only 的受控 binary/subprocess。
- 第一阶段优先实现 `/v1/images/*` 与 `/v1/videos/*`。
- OpenAI Codex 与 ChatGPT Web session 必须拆开。
- Gemini CLI OAuth、Gemini Web session、Antigravity 必须拆开。

## 最终采用清单

用户最终采用：

```text
OAI-WEB-01
OAI-CODEX-04
GEM-CLI-02
GEM-WEB-01
```

## 关键判断

### `OAI-CODEX-04` 的限制

`OAI-CODEX-04` 是 `codex-manager`，它的价值在账号控制面：

- Codex 账号导入。
- OAuth/token 刷新。
- 账号验证。
- 导出 Sub2API/CPA 格式。
- 多账号资源管理。

它不是 `gpt-image-2` 执行内核。因此第一阶段如果只采用 `OAI-CODEX-04`，`openai_codex` 只能先完成账号管理与控制面，不能直接承诺 `/v1/images/generations` 可用。

后续如果要让 Codex 订阅真正执行生图，需要在下一轮追加执行内核，或由平台参考 Codex manager 的账号结构重写执行层。

### `GEM-CLI-02` 的限制

`GEM-CLI-02` 是通用 CLI/Agent proxy，适合统一处理 Gemini CLI OAuth/profile、账号轮询、健康检查和 OpenAI-compatible bridge。

但 Gemini CLI 路线不应直接等同于 Gemini Web 的图片/视频生成能力。第一阶段把 `GEM-CLI-02` 定型为账号/Agent 执行通道，不把它标为 `/v1/images/*` 或 `/v1/videos/*` 的强承诺内核。

### `GEM-WEB-01` 的定位

`GEM-WEB-01` 是 Gemini Web session 路线。它与 Gemini CLI OAuth 必须拆开：

- `gemini_cli_oauth`：面向 CLI OAuth/profile。
- `gemini_web_session`：面向 Gemini Web cookie/session。

Gemini 图片/视频优先走 `gemini_web_session`，因为 Web 产品形态更接近真实 image/video 生成工作流。

### `AG-01` 的定位

`AG-01` 只归入 `antigravity`，不得和 Gemini CLI 或 Gemini Web 混用。

第一阶段优先完成：

- Antigravity 账号导入。
- 本地 profile / OAuth / session 识别。
- 代理调用。
- 诊断与 health check。

媒体能力不在第一阶段强承诺。

## 为什么没有包含 Qwen 生图 / 生视频

Qwen 本轮不进入定型，不是因为 Qwen 没有图像或视频生态，而是因为它不符合这次的反代内核准入条件。

核验结果：

- Qwen Code 类项目主要是 coding agent、CLI、OpenAI-compatible 文本/工具调用代理，不是生图/生视频执行内核。
- 已检索到的 `qwen-code-oai-proxy` 类项目明确偏 OpenAI-compatible 文本代理，并且存在免费额度失效/维护风险。
- Qwen-Image 是开源生图模型，适合自托管推理，不是 Web/session/CLI 反代内核。
- Wan2.1/Wan2.2 是开源视频模型，适合自托管推理，不是 Qwen Web session 反代内核。
- Wan2GP 这类项目能跑 Qwen Image / Wan / Flux / HunyuanVideo 等本地模型，但属于 `SELF` 自托管 worker 路线；本轮用户已声明 `SELF：无可用`。

因此 Qwen 生图/生视频如果未来要做，只能走两条新路线：

1. 找到稳定的 Qwen Web / 通义 Web session 生图生视频反代仓库。
2. 重新开放 `SELF` 自托管路线，把 Qwen-Image、Wan2.x、Wan2GP、ComfyUI 作为本地推理 worker。

在当前硬性前提下，Qwen 标记为“无可用”是合理的。

## 为什么没有单独包含豆包生图 / 生视频

豆包没有作为独立 provider 进入本轮定型，原因分两层：

第一，豆包生态中与媒体生成最接近的是 Jimeng/Dreamina/Seedream/Seedance 方向。本轮已经采用 `JM-01`：

```text
JM-01 -> jimeng_web_session -> iptag/jimeng-api
```

它覆盖的是 Jimeng/Dreamina 图片逆向，属于豆包/字节媒体生态里更贴近反代要求的部分。

第二，检索到的“豆包生图/生视频”项目大多属于以下几类：

- Volcengine / Ark / Seedream / Seedance API key。
- MCP server 包装官方或半官方 API。
- AceDataCloud 等第三方聚合 API。
- 教程或手工浏览器流程。
- 新出现的小型 reverse-engineered 项目，成熟度和接口稳定性还需要单独审计。

这些不满足本轮“禁止官方 SDK/API、优先稳定反代内核”的要求。

因此本轮处理方式是：

- 不新增 `doubao_web_session`。
- 用 `JM-01` 先覆盖 Jimeng/Dreamina 图片方向。
- Seedance/豆包视频后续需要独立补一轮 `doubao_web_session` 或 `seedance_web_session` 调研和审计。

## 为什么没有包含文心一言生图 / 生视频

文心一言没有进入本轮定型，原因是没有找到满足条件的稳定 Web/session 媒体反代内核。

核验结果：

- 文心一言/ERNIE 相关开源更多是官方模型、SDK、云 API 或应用集成。
- `ERNIE-Image` 是百度开源的文生图模型，适合自托管推理，不是文心一言 Web session 反代。
- 没有找到稳定覆盖“文心一言 Web session -> `/v1/images/*` / `/v1/videos/*`”的成熟开源仓库。
- 文心视频生成方向更偏官方云产品、第三方聚合或通用视频模型，不符合本轮反代内核条件。

如果未来要纳入文心一言，需要重新寻找：

1. 文心一言 Web session 反代仓库。
2. ERNIE Bot / ERNIE Image 非官方 session connector。
3. 或重新开放 `SELF` 路线，把 `ERNIE-Image` 作为本地推理 worker。

当前本轮不纳入是合理的。

## 最低验收标准

每个进入“可用”状态的 provider 必须满足：

1. 账号材料可导入，敏感信息密文保存。
2. health check 能区分未登录、失效、限流、额度不足、接口变化、生成失败。
3. 图片 provider 支持 `/v1/images/generations`，编辑类 provider 支持 `/v1/images/edits`。
4. 视频 provider 支持 `/v1/videos` 创建任务和 `/v1/videos/{id}` 查询任务。
5. 生成产物统一入 `media_assets`。
6. Web/session 类账号默认并发 1。
7. 不跨 provider 复用账号材料。
8. 不在 UI 暴露上游晦涩字段。
9. 若使用 subprocess，必须固定版本/hash、只监听 loopback、可停止、可观测。
