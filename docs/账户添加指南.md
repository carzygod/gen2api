# gen2api 账户添加指南

文档版本：v1.1
编写日期：2026-06-09
适用范围：图片/视频生成版 `sub2api` 平台的 Web Cookie Provider、Agent Provider、账号池和授权资源接入。

## 1. 核心原则

`gen2api` 主平台的核心职责是管理 Web Cookie/session 与 Agent Provider credential/profile，并围绕这些资源完成账号池、调度、计费、资产、验收和审计。外部 connector、sidecar、Vault 或自托管服务是可选执行层；首期产品不应把 Web Cookie 和 Agent Provider 排除在主平台之外。

不要在主平台中长期明文保存：

- 上游账号密码。
- 浏览器 cookie 明文。
- 验证码处理流程。
- 绕过风控的脚本或说明。
- 上游第一方控制台 API key 获取教程。

可以在主平台中保存的账号资源只分两类：

- `web_cookie_provider + cookie_secret`：加密后的 cookie/session secret，引用通常是 `secret://...`。
- `agent_provider + agent_provider_credential`：加密后的 Agent Provider credential/profile，或外部 `agent://...` 引用。
- 旧的 OAuth/CLI/MCP/token/subscription 字段只作为兼容导入来源，导入后必须归一到上面两类。
- 已存在的 `secret://...` 会按平台 secret 的真实 `kind` 校验资源类型：`kind=cookie` 只能绑定 Web Cookie Provider，`kind=agent_provider` 只能绑定 Agent Provider，不能靠表单声明跨类型复用。
- `secret://...` 不是免校验通道；如果 provider 在 `platform-input-requirements` 声明了必填凭据字段，平台会解密 active secret 并按真实字段名复验。缺少 `LUMA_API_KEY`、`GEMINI_CREDENTIALS` 等字段，或用 `plain://raw-token` 这类无字段名 inline 引用冒充托管材料，都会返回 `PROVIDER_REQUIRED_INPUT_MISSING`。

## 2. 推荐操作流程

管理台路径：

1. 打开 `/admin` 并登录。
2. 进入“授权资源”，先选择目标 provider。
3. 在“平台”里选择目标 provider，点击“查看平台指南”。
4. 只在 `Web Cookie` 与 `Agent Provider` 两个标签里选择一个。
5. 如果是 Web Cookie provider，进入“Web Cookie”，粘贴 cookie header、JSON cookie jar、浏览器导出片段或 `secret://...` 引用；保存时平台必须转成加密 secret/ref。
6. 如果是 Agent Provider，进入“Agent Provider”，填写 CLI/Agent profile、credential cache、MCP config、runtime endpoint 或 `agent://...` 引用。
7. 如果对应项目支持 delegated OAuth、Web session assist 或 Agent Provider 授权助手，进入“授权会话”；授权完成后仍要回写成 Cookie 或 Agent Provider，执行器地址只有在该项目明确要求服务地址时才填写。
8. 如果对应项目已经直接返回 `credential_ref`，也可以进入“授权资源”或“账号池”，点击“添加平台账号”。
9. 粘贴对应项目返回的 `secret://`、`agent://` 或托管 JSON；旧字段名只作为兼容导入，不作为新增账号类型。
10. 保存后运行“账号验收套件”或对应账号的 external acceptance。

最短路径是“授权资源”里的 Web Cookie 或 Agent Provider 表单。旧的“一键准备账号”和批量导入只用于把已有资源清单归一成这两类账号。

```powershell
curl -X POST `
  -H "Authorization: Bearer dev-admin-key" `
  -H "Content-Type: application/json" `
  http://127.0.0.1:8080/v1/admin/account-setup-quickstart `
  -d "{
    \"provider_id\":\"gemini\",
    \"resource_type\":\"agent_provider\",
    \"credential_ref\":\"agent://providers/gemini/acct_01\",
    \"resource_profile\":{\"workspace_policy\":\"isolated\"},
    \"account_id\":\"acct_gemini_01\",
    \"operations\":[\"text_to_image\",\"image_edit\",\"text_to_video\",\"image_to_video\"],
    \"supported_provider_models\":[\"nano-banana-pro\",\"veo-3.1\"],
    \"dry_run\":true
  }"
```

先用 `dry_run=true` 看 `steps` 和 `action_items`；确认后改成 `dry_run=false`。如果返回授权会话，先完成兼容授权会话，再确认最终账号是 `web_cookie_provider` 或 `agent_provider`。

### 2.1 刷新开源 connector registry

```powershell

  -H "Authorization: Bearer dev-admin-key" `
  -H "Content-Type: application/json" `
  http://127.0.0.1:8080/v1/admin/connector-registry/refresh `
  -d "{}"
```

```powershell
curl -H "Authorization: Bearer dev-admin-key" `
  "http://127.0.0.1:8080/v1/admin/connector-registry?provider_id=gemini"
```

### 2.2 获取 provider 添加指南

查看全部 provider 指南：

```powershell
curl -H "Authorization: Bearer dev-admin-key" `
  http://127.0.0.1:8080/v1/admin/account-guides
```

查看某个平台，例如 Gemini：

```powershell
curl -H "Authorization: Bearer dev-admin-key" `
  http://127.0.0.1:8080/v1/admin/account-guides/gemini
```

返回内容会包含：

- 推荐资源类型。
- 推荐鉴权方式。
- 可选执行器地址示例；仅在对应开源项目明确要求服务地址时使用。
- `input_requirements` 专属字段；账号向导会把非凭据、非执行器地址的字段动态渲染成表单，例如 Midjourney 的 `guild_id` / `channel_id`。
- `input_requirements[].evidence` 字段证据；用于说明该输入来自哪个本地开源项目 README、配置示例或节点定义。
- `credential_ref` 示例。
- 支持 operations。
- 支持 provider models。
- 可直接修改的 onboarding payload。
- 相关 GitHub 开源项目。

### 2.4.1 发起授权会话

如果对应开源项目支持 OAuth start/status、Web session assist 或 Agent Provider 授权助手，可以从主平台创建授权会话。主路径是 `/authorized-resource-sessions`，旧 `/oauth-sessions` 仅作为兼容路径，产品含义统一是“授权资源会话”：

下面示例默认不包含 `provider_base_url`。只有 `/v1/admin/platform-input-conformance` 显示 `runtime_base_url_allowed=true`，且你实际使用的开源项目要求外部 wrapper/sidecar 地址时，才额外加入该字段。

```powershell
curl -X POST `
  -H "Authorization: Bearer dev-admin-key" `
  -H "Content-Type: application/json" `
  http://127.0.0.1:8080/v1/admin/authorized-resource-sessions `
  -d "{
    \"provider_id\":\"gemini\",
    \"account_id\":\"acct_gemini_01\",
    \"label\":\"Gemini production account 01\",
    \"auth_method\":\"agent_provider_credential\",
    \"resource_type\":\"agent_provider\",
    \"supported_operations\":[\"text_to_image\",\"image_edit\",\"text_to_video\",\"image_to_video\"],
    \"supported_provider_models\":[\"nano-banana-pro\",\"veo-3.1\"]
  }"
```

返回 `authorize_url` 时，在对应项目控制的页面完成授权。主平台不会长期保存上游 OAuth、cookie 或 agent profile 明文材料。

授权完成后，从对应项目回收 `credential_ref` 并自动创建账号：

```powershell
curl -X POST `
  -H "Authorization: Bearer dev-admin-key" `
  -H "Content-Type: application/json" `
  http://127.0.0.1:8080/v1/admin/authorized-resource-sessions/{session_id}/complete `
  -d "{
    \"create_account\":true,
    \"sync_capabilities\":true,
    \"run_health_check\":true
  }"
```

如果对应项目已经把引用返回给你，也可以手动完成：

```powershell
curl -X POST `
  -H "Authorization: Bearer dev-admin-key" `
  -H "Content-Type: application/json" `
  http://127.0.0.1:8080/v1/admin/authorized-resource-sessions/{session_id}/complete `
  -d "{
    \"credential_ref\":\"agent://providers/gemini/acct_01\",
    \"auth_method\":\"agent_provider_credential\",
    \"resource_type\":\"agent_provider\",
    \"create_account\":true
  }"
```

如果对应项目返回的是一次性凭据材料，而不是已经托管好的 `credential_ref`，可以传 `credential_value` 或在 `connector_response` 内传 `credential_value` / `credential_material`。该字段必须使用对应开源项目真实字段名；例如 Runway/UseAPI 使用 `apiKey`，Luma 使用 `LUMA_API_KEY`，Jimeng/Seedream 使用 `api_key` 或 `apiKey`。平台会先按字段校验，再加密托管成 `secret://...`，响应和会话记录不会回显明文：

```powershell
curl -X POST `
  -H "Authorization: Bearer dev-admin-key" `
  -H "Content-Type: application/json" `
  http://127.0.0.1:8080/v1/admin/authorized-resource-sessions/{session_id}/complete `
  -d "{
    \"credential_value\":{\"apiKey\":\"useapi-key\"},
    \"auth_method\":\"agent_provider_credential\",
    \"resource_type\":\"agent_provider\",
    \"supported_operations\":[\"text_to_video\"],
    \"supported_provider_models\":[\"runway-gen4\"],
    \"create_account\":true
  }"
```

如果对应项目返回的是完整 JSON，直接粘贴到管理台“完成 JSON”，或通过 API 传入 `connector_response`。平台会自动读取 `account.credential_ref`、`account.resource_type`、`account.resource_profile`、`account.supported_operations`、`account.supported_provider_models`、`quota_buckets`、`concurrency_limit`、`region` 和 `plan`：

```powershell
curl -X POST `
  -H "Authorization: Bearer dev-admin-key" `
  -H "Content-Type: application/json" `
  http://127.0.0.1:8080/v1/admin/authorized-resource-sessions/{session_id}/complete `
  -d "{
    \"connector_response\":{
      \"status\":\"completed\",
      \"account\":{
        \"id\":\"acct_gemini_01\",
        \"label\":\"Gemini production account 01\",
        \"credential_ref\":\"agent://providers/gemini/acct_01\",
        \"auth_method\":\"agent_provider_credential\",
        \"resource_type\":\"agent_provider\",
        \"resource_profile\":{\"workspace_policy\":\"isolated\"},
        \"supported_operations\":[\"text_to_image\",\"text_to_video\"],
        \"supported_provider_models\":[\"nano-banana-pro\",\"veo-3.1\"],
        \"quota_buckets\":[{\"type\":\"credits\",\"remaining_estimate\":200}],
        \"concurrency_limit\":2,
        \"plan\":\"pro\"
      }
    },
    \"create_account\":true
  }"
```

授权助手也可以在登录完成后主动回调：

```powershell
curl -X POST `
  -H "Authorization: Bearer dev-admin-key" `
  -H "Content-Type: application/json" `
  http://127.0.0.1:8080/v1/admin/authorized-resource-sessions/{session_id}/callback `
  -d "{\"status\":\"completed\",\"credential_ref\":\"agent://providers/gemini/acct_01\"}"
```

### 2.4.2 资源清单一键导入账号池

当 sub2api 类项目、Web-to-API sidecar、CLIProxy/AIClient2API 或自有脚本已经输出账号资源清单时，不需要逐条手动添加。进入 `/admin` 后打开“授权资源”或“账号池”区域下的“批量导入账号”，填写：

- `平台`：例如 `gemini`、`kling`、`jimeng`、`runway`。
- `鉴权方式`：只选 `cookie_secret` 或 `agent_provider_credential`。
- `资源清单 URL`：connector 或自有脚本暴露的账号清单地址，可留空。
- `可选执行器 Base URL`：仅在对应项目要求 runner/sidecar 服务地址时填写。
- `账号 JSON / JSONL`：可粘贴 JSON 数组、JSONL，或带 `accounts` / `resources` / `items` / `data` 字段的包装对象。

也支持更接近 sub2api、CLIProxyAPI、AIClient2API、Web-to-API sidecar 的嵌套导出格式。平台会自动识别 `provider/platform`、`auth.type`、`auth.ref`、`connectorBaseUrl`、`models[].operations`、`quota.remaining`、`maxConcurrency` 等字段：

```json
{
  "accounts": {
    "gemini_sub2api_01": {
      "provider": "gemini",
      "name": "Gemini Agent from sub2api export",
      "auth": {
        "type": "agent",
        "ref": "agent://providers/gemini/sub2api_01"
      },
      "models": [
        {"id": "nano-banana-pro", "operations": ["t2i", "edit"]},
        {"id": "veo-3.1", "capabilities": ["t2v", "i2v"]}
      ],
      "quota": {"type": "credits", "remaining": 120},
      "maxConcurrency": 2,
      "region": "global",
      "tier": "pro"
    }
  }
}
```

先预览，不写数据库：

下面的 `provider_base_url` 仍然只适用于 Gemini 外部 agent wrapper 这类 `runtime_base_url_allowed=true` 的平台；例如 Midjourney 这类不要求 runner/baseURL 的平台，预览和导入请求都会拒绝该字段。

```powershell
curl -X POST `
  -H "Authorization: Bearer dev-admin-key" `
  -H "Content-Type: application/json" `
  http://127.0.0.1:8080/v1/admin/account-subscriptions/preview `
  -d "{
    \"provider_id\":\"gemini\",
    \"auth_method\":\"agent_provider_credential\",
    \"content\":\"{\\\"accounts\\\":[{\\\"account_id\\\":\\\"acct_gemini_01\\\",\\\"label\\\":\\\"Gemini 01\\\",\\\"credential_ref\\\":\\\"agent://providers/gemini/acct_01\\\",\\\"resource_type\\\":\\\"agent_provider\\\",\\\"concurrency_limit\\\":1}]}\"
  }"
```

确认预览中的 `planned`、`warnings`、`failed` 和模型能力后执行导入：

```powershell
curl -X POST `
  -H "Authorization: Bearer dev-admin-key" `
  -H "Content-Type: application/json" `
  http://127.0.0.1:8080/v1/admin/account-subscriptions/import `
  -d "{
    \"provider_id\":\"gemini\",
    \"auth_method\":\"agent_provider_credential\",
    \"content\":\"{\\\"accounts\\\":[{\\\"account_id\\\":\\\"acct_gemini_01\\\",\\\"label\\\":\\\"Gemini 01\\\",\\\"credential_ref\\\":\\\"agent://providers/gemini/acct_01\\\",\\\"resource_type\\\":\\\"agent_provider\\\",\\\"concurrency_limit\\\":1}]}\",
    \"sync_capabilities\":false,
    \"run_health_check\":false
  }"
```

正式导入响应会返回 `summary.created`、`summary.failed` 和每个账号的 onboarding 结果。默认情况下，正式导入账号会自动补齐模型映射：平台根据 `models[].operations`、`supported_operations` 和 `supported_provider_models` 创建缺失的 `ProviderModelMapping`。例如 `t2i` 会映射到 `t2i-fast`，`edit` 会映射到 `image-edit`，`t2v` 会映射到 `t2v-general`，`i2v` 会映射到 `i2v-fast`。如果你希望只导入账号、不改映射，在导入请求中设置 `"auto_create_mappings": false`。

支持的清单字段：

| 字段 | 说明 |
| --- | --- |
| `account_id` / `id` | 账号资源 ID |
| `label` / `name` / `displayName` | 后台展示名 |
| `credential_ref` / `credentialRef` / `token_reference` / `oauth_ref` / `webSessionRef` / `accountRef` / `vaultRef` | connector 返回的安全凭据引用 |
| `auth.type` / `authType` / `auth_method` | 鉴权方式；最终只归一为 `cookie_secret` 或 `agent_provider_credential`，`oauth`、`web_session`、`cli`、`subscription`、`mcp`、`endpoint` 等仅是兼容别名 |
| `subscription_url` / `subscriptionUrl` | 资源清单地址；仅用于拉取账号清单，不作为账号凭据本体保存，清单中的账号仍必须给出 cookie/session 或 agent profile 引用 |
| `provider_base_url` / `connector_base_url` / `connectorBaseUrl` / `baseUrl` | 可选执行器地址；仅对应项目要求 runner/sidecar 服务地址时使用 |
| `supported_operations` / `operations` / `models[].operations` | 账号支持的操作；支持 `t2i`、`edit`、`t2v`、`i2v` 等别名 |
| `supported_provider_models` / `provider_models` / `models[].id` | 账号支持的上游模型；缺省时使用源码 provider 模板默认值 |
| `quota_buckets` / `quota` / `remainingCredits` | 额度桶 |
| `concurrency_limit` / `maxConcurrency` | 并发上限 |
| `region` / `plan` / `tier` / `status` | 资源标签和状态 |

### 2.4.3 创建长期资源清单源并同步账号池

一次性导入适合临时粘贴。生产环境建议创建长期资源清单源，让平台后续可以重复预览和同步账号池。旧 API 字段仍叫 `subscription_url`，但产品含义是“资源清单 URL”，不是一种账号鉴权方式。

管理台路径：

1. 打开 `/admin`。
2. 进入“授权资源”区域里的“批量导入账号”面板。
3. 选择平台和鉴权方式。
4. 填写项目暴露的“资源清单 URL”；只有项目要求服务地址时才填写“可选执行器 Base URL”。
5. 填写“资源清单源名称”，点击“创建资源清单源”。
6. 系统返回 `source.id` 后会自动写入“资源清单源 ID”。
7. 选择“预览同步”，点击“同步资源清单源”，确认 planned、failed、warnings。
8. 选择“正式同步”，再次点击“同步资源清单源”，账号会进入账号池。

创建资源清单源：

如果目标平台的 `/v1/admin/platform-input-conformance` 显示 `runtime_base_url_allowed=false`，创建资源清单源时也必须删除 `provider_base_url`。

```powershell
curl -X POST `
  -H "Authorization: Bearer dev-admin-key" `
  -H "Content-Type: application/json" `
  http://127.0.0.1:8080/v1/admin/account-subscription-sources `
  -d "{
    \"provider_id\":\"gemini\",
    \"name\":\"Gemini resource list pool\",
    \"auth_method\":\"agent_provider_credential\",
    \"subscription_url\":\"https://resource-list.example/gemini/accounts.json\",
    \"max_items\":200
  }"
```

预览同步，不写账号池：

```powershell
curl -X POST `
  -H "Authorization: Bearer dev-admin-key" `
  -H "Content-Type: application/json" `
  http://127.0.0.1:8080/v1/admin/account-subscription-sources/{source_id}/sync `
  -d "{\"dry_run\":true}"
```

正式同步：

```powershell
curl -X POST `
  -H "Authorization: Bearer dev-admin-key" `
  -H "Content-Type: application/json" `
  http://127.0.0.1:8080/v1/admin/account-subscription-sources/{source_id}/sync `
  -d "{\"dry_run\":false}"
```

查看资源清单源：

```powershell
curl -H "Authorization: Bearer dev-admin-key" `
  http://127.0.0.1:8080/v1/admin/account-subscription-sources
```

### 2.4.4 资源清单源生产化预演与验收

资源清单源同步只是把 Web Cookie / Agent Provider 账号放入账号池。进入生产路由前，还需要能力同步、健康检查、额度检查和 connector preflight。管理台中可以在同一个“批量导入账号”面板填写“资源清单源 ID”，点击“生产化预检”。

推荐顺序：

1. “同步模式”选择“预览同步”，点击“生产化预检”，只看计划和缺口。
2. 修复返回的 `action_items`，例如对应项目要求执行器地址但未填写、缺少 mapping、缺少 quota endpoint。
3. “同步模式”选择“正式同步”，再次点击“生产化预检”，执行真实同步、能力同步、健康检查、额度检查和 preflight。
4. 如果要消耗真实额度验收，设置 `run_acceptance=true` 和 `run_samples=true`，最多先跑 1 条样本。

## 3. 鉴权方式说明

产品主路径只保留两类账号资源。其它字段只作为历史导入别名、开源项目原始字段或执行层配置解析，入库后必须归一到下表两类之一。

| `auth_method` | 适用场景 | 填写建议 |
| --- | --- | --- |
| `cookie_secret` | Web cookie/session、JSON cookie jar、浏览器导出片段、网页登录态刷新结果 | 一次性粘贴后由平台加密为 `secret://...`，或直接填写已存在的 `secret://...` |
| `agent_provider_credential` | Codex/Gemini/Qwen/Grok 等 Agent Provider profile、CLI credential cache、agent runtime config、MCP/agent 配置引用 | 优先填写 `agent://providers/{provider}/acct_01`，需要托管时可提交一次性 profile 并转为 `secret://...` |

兼容归一规则：

| 兼容输入 | 归一结果 |
| --- | --- |
| `web_session_reference`、`websession://...`、cookie header、cookie jar | `cookie_secret` |
| `oauth_reference`、`cli_credential_reference`、`mcp_config_reference`、`secret_json`、agent runtime endpoint | `agent_provider_credential` |
| `subscription_url`、`subscription_content` | 只表示资源清单来源，不是账号鉴权；清单内的每个账号仍要归一为 `cookie_secret` 或 `agent_provider_credential` |
| `self_hosted_endpoint`、`provider_base_url`、`connector_base_url` | 只在对应开源项目要求外部 runner/sidecar 时作为执行层地址保存 |
| `api_key`、`bearer_token`、`aggregator_api_key` | 不作为新账号资源方式；如旧导入中出现，只能作为 Agent Provider/执行层兼容材料处理，并应避免在用户表单中暴露 |

## 4. 生产启用条件

账号进入生产路由前必须满足：

1. Provider 状态为 `active`。
2. Account 状态为 `active`。
3. `credential_ref` 可用。
4. 支持的 operation 和 provider model 与 mapping 对齐。
5. health check 通过。
6. capabilities sync 通过。
7. quota sync 或预算控制可用。
8. external acceptance 至少一个真实样本成功。
9. 输出图片/视频已转存为平台 `MediaAsset`。
10. Web Cookie provider 已做加密存储、域名 allowlist、过期检测、人工审批和并发限制。
11. Agent Provider 已做 runtime 隔离、工作目录隔离、credential 脱敏和健康检查。
12. 对开源项目画像声明为必填的字段，所有账号入口都会按 `platform-input-requirements` 强制校验，授权会话的“完成/回调”也不能绕过；例如 Midjourney 必须同时提交 Discord 会话材料与 `guild_id`、`channel_id`，缺失时返回 `PROVIDER_REQUIRED_INPUT_MISSING`。

## 授权会话启动字段传递补充

授权会话的“发起会话”阶段也必须按 `/v1/admin/platform-input-requirements` 传递平台专属 `resource_profile` 字段。比如 Midjourney 类 Web Cookie Provider 在 start payload 中就应带上 `guild_id`、`channel_id` 等 open-source profile 字段；Gemini/Qwen 类 Agent Provider 则带 agent profile/runtime 相关字段。`/v1/admin/authorized-resource-sessions` 会把 `resource_type` 与 `resource_profile` 原样传给对应 sidecar/agent helper，`/complete` 与 `/callback` 仍会再次合并和校验这些字段，防止空 profile 或泛化 baseURL 绕过真实平台要求。

## 批量导入动态示例补充

批量导入的输入示例也必须按当前 `provider + auth_method` 从 `/v1/admin/platform-input-requirements` 动态生成。凭据字段来自带 `auth_method` 或 `store_as` 的条目，平台专属画像字段来自非凭据、非执行器地址条目并写入 `resource_profile`。例如 Midjourney 示例必须出现 Discord 会话材料、`guild_id`、`channel_id`，但不出现通用 `base_url`；Gemini/Qwen 示例应体现 Agent profile 或 credential cache，可选 runtime endpoint 只有在开源项目要求外部 runner/sidecar 时出现。批量预览、正式导入和长期资源清单源同步都必须复用同一后端校验。

## 凭据输入字段动态提示补充

管理台的 Web Cookie、Agent Provider 和“添加平台账号”向导不再只显示泛化的“授权材料”说明。页面会从 `/v1/admin/platform-input-requirements` 的 `user_inputs` 中筛出带 `auth_method` 或 `store_as` 的凭据字段，并按当前平台与鉴权方式刷新输入框标题、占位示例和证据提示。例如 Midjourney 会提示 `Discord/Midjourney session 或 user token`，Gemini 会提示 `GEMINI_CREDENTIALS`、`GOOGLE_APPLICATION_CREDENTIALS`、`gemini_oauth_creds_file` 等 one-of 字段。多个凭据字段可以作为 JSON 一次性提交，平台只保存 `secret://...` 或 `agent://...` 引用。

Kling 这类开源项目明确要求成对密钥的平台不得再合并成一个泛化字段。直接粘贴 `credential_value` 时应提交 JSON 或 `.env` 形态，例如同时包含 `KLING_ACCESS_KEY` 与 `KLING_SECRET_KEY`；如果提交的是 `agent://...` 或 `secret://...` 托管引用，则由对应 Agent Provider/secret 负责持有这组完整材料。

Gemini/Qwen 这类 CLI OAuth 平台按开源项目的真实替代字段展示为 one-of 组。Gemini 直接粘贴时至少提供 `GEMINI_CREDENTIALS`、`GOOGLE_APPLICATION_CREDENTIALS`、`gemini_oauth_creds_base64`、`gemini_oauth_creds_file` 之一，`GEMINI_PROJECT_ID`/`gemini_project_id` 只在所选 wrapper 要求显式 project id 时补充；Qwen 至少提供 `qwen_oauth_creds_file`、`qwen_oauth_cache_path` 或 `qwen_oauth_credentials` 之一。只提供 project id、runtime endpoint 或通用 base URL 不算有效账号材料。

其它 Agent Provider/聚合型平台也必须按对应开源仓库的真实字段名提交，不能再用一个无名 token 混过去。Jimeng/Seedream 按 ComfyUI-Jimeng-API/seedance-api 使用 `apiKey`/`api_key`；Luma 按 luma-ai-mcp-server 使用 `LUMA_API_KEY`；Runway 的 UseAPI 路径使用 `apiKey`，只有执行 UseAPI 注册 helper 时才把 `runwayEmail`、`runwayPassword` 放入加密 Agent Provider 材料；Pollinations 使用 `POLLINATIONS_KEY` 或 `POLLINATIONS_API_KEY`；OpenRouter 使用 `OPENROUTER_API_KEY`，或代理项目里的 `OPENROUTER_API_KEY_N`、`ANTHROPIC_AUTH_TOKEN`、`ANTHROPIC_API_KEY`；fal/Replicate 使用 `FAL_KEY` 或 `REPLICATE_API_TOKEN`；Flux/Stability 本地后端使用 ComfyUI workflow API JSON、FastFusion `model_config.json` 或 MeiGen MCP config。

这些精确字段在直接粘贴 `credential_value` 时应使用 JSON 或 `.env` 形态，例如 `{"LUMA_API_KEY":"..."}` 或 `FAL_KEY=...`。提交 `secret://...` 引用时，平台会读取 active secret 的真实密文并按同名字段复验；不存在、非 active 或只有 raw token 的 secret 不会通过。提交 `agent://...` 引用时，平台把它视为外部 Agent Provider profile 引用，由该 agent/profile 保证同名字段完整。
